# 작업 맥락 (Work Context)

이 파일은 테스트 자동화 툴 개발 과정에서 나누는 대화의 맥락을 기록하고 보존하기 위해 만들어졌습니다.

---

## 실행 방법 (How to Run)

**중요:** 이 애플리케이션은 Mac의 Automator로 만든 실행 앱을 통해 시작하는 것이 가장 편리하고 안정적입니다.

**실행 순서:**

1.  프로젝트 폴더에 포함된 (또는 직접 생성한) **'QUATY 실행' Automator 앱** 아이콘을 더블클릭합니다.
2.  나타나는 'QUATY Updater & Launcher' 창에서 **'Update & Launch QUATY'** 버튼을 클릭합니다.

실행기는 자동으로 GitHub에서 최신 코드를 받아온 후, 메인 애플리케이션을 실행합니다.

*참고: Automator 앱 생성 방법은 '대화 13' 기록을 참조하세요.*

## 미래 개발 로드맵 (Future Development Roadmap)

**최종 목표:** 사용자가 제공한 'QUATY Test Automation Tool' 스크린샷을 청사진으로 삼아, UI와 기능 면에서 동일한 수준의 전문적인 툴을 구현합니다. API 사용 효율을 높이기 위해 다음의 단계별 계획에 따라 체계적으로 개발을 진행합니다.

### Phase 1: UI 기반 구축

1.  **Dark Theme 적용:** 앱 전체에 'QUATY'와 유사한 어두운 테마의 스타일시트를 적용하여 시각적 정체성을 확립합니다. (**완료**)
2.  **레이아웃 재구성:** 'QUATY'의 레이아웃을 참조하여, 화면을 크게 **좌측 패널 (환경 설정, 실행 옵션 등)**, **중앙 패널 (URL, 스텝 테이블)**, **하단 패널 (실행 로그)**의 3단 구조로 재구성합니다. (**완료**)
3.  **위젯 재배치:** 기존의 위젯들을 새로운 레이아웃 구조에 맞게 재배치합니다. (**완료**)

### Phase 2: 핵심 기능 강화

1.  **Assertion Mode 구현:** 테스트 케이스에 '검증(assertion)' 스텝을 추가할 수 있는 기능을 구현합니다. (**기록 기능 구현 완료**)
2.  **수동 스텝 추가 (+ Step):** 녹화 외에 사용자가 직접 스텝을 추가할 수 있는 기능을 구현합니다.
3.  **Headless 모드:** GUI 없이 백그라운드에서 테스트를 실행할 수 있는 옵션을 추가합니다.
4.  **스텝 선택 (Select) 기능:** 테이블에 체크박스를 추가하여 특정 스텝들을 선택하고 일괄 작업을 수행할 수 있는 기반을 마련합니다.

### Phase 3: 고급 기능 및 편의성 개선

1.  **CSV 데이터 연동:** 외부 CSV 파일의 데이터를 변수처럼 테스트에 주입하여 데이터 기반 테스트가 가능하도록 구현합니다.
2.  **All Batch 실행:** 파일 탐색기에서 여러 테스트 케이스를 선택하여 순차적으로 한 번에 실행하는 기능을 구현합니다.
3.  **환경 설정 (Environment):** 'Alpha', 'Real' 등 테스트를 실행할 서버 환경을 선택할 수 있는 옵션을 추가합니다.
4.  **Save Flow 기능:** 현재의 'Save Test' 기능을 대체하거나 보강하여, 더 직관적인 이름으로 테스트 흐름 전체를 저장하는 기능을 구현합니다.

## 대화 기록 (Dialogue History)

### 대화 1 ~ 11 (요약)
- 프로젝트 시작부터 UI 개선 착수까지의 과정을 기록함.

### 대화 12: 레이아웃 재구성 및 실행기 도입

**USER:**
- 채도를 낮춘 버튼 색상 수정 및 3단 레이아웃 재구성 작업을 요청.
- 앱이 깨져도 안전하게 업데이트할 수 있는 더 편리한 실행 방법을 문의.

**GEMINI:**
- \`stylesheet.qss\`의 버튼 색상 조정, \`main.py\`의 3단 레이아웃 재구성 완료.
- 메인 앱과 분리된 별도의 '업데이트 실행기'(\`launcher.py\`)를 제안하고 구현. 이 실행기는 \`git pull\`로 코드를 최신화한 후 \`main.py\`를 실행시켜, 앱 자체에 오류가 발생해도 안전하게 업데이트할 수 있도록 함.

### 대화 13: Automator를 이용한 원클릭 실행 구현

**USER:**
- \`launcher.py\` 실행 시 \`git pull\` 충돌 문제 발생.
- 문제를 해결한 후, 터미널 명령어 대신 Mac의 Automator를 활용하여 앱을 실행하는 아이디어를 제안.
- Automator 실행 시 \`ModuleNotFoundError: No module named 'PySide6'\` 오류 발생.
- 최종적으로 \`launcher.py\`가 메인 앱을 실행하지 못하는 문제 발생.

**GEMINI:**
- \`git pull\` 충돌은 로컬의 추적되지 않는 \`launcher.py\` 파일과 원격 저장소의 파일이 충돌하여 발생. \`git pull origin main\`을 수동으로 한 번 실행하여 해결.
- Automator의 \`ModuleNotFoundError\`는 시스템 기본 파이썬 경로 문제임을 파악. \`which python3\`로 정확한 파이썬 경로를 찾아 Automator 스크립트에 명시하는 방법으로 해결.
- \`launcher.py\`가 \`main.py\`를 실행 못하는 문제는 \`subprocess.Popen\`에서 파이썬 경로를 잃어버리기 때문임을 파악. \`launcher.py\`가 \`main.py\`를 호출할 때, \`sys.executable\` 변수를 사용하여 자신을 실행시킨 파이썬 경로를 그대로 넘겨주도록 코드를 수정하여 최종 해결.
- 모든 과정을 거쳐, 아이콘 더블클릭만으로 업데이트와 실행이 모두 가능한 완전한 워크플로우를 구축하고, 해당 내용을 \`CONTEXT.md\`에 최종 반영함.

### 대화 14: Assertion Mode 기록 기능 구현

**USER:**
- Phase 2의 첫 단계로 'Assertion Mode' 기능 구현을 요청.
- 더 나은 UX를 위해 버튼 대신 체크박스를 제안하고, 녹화 시작 전에 모드를 설정하는 워크플로우를 제시.
- 검증 모드에서 브라우저의 클릭 이벤트가 동작하지 않는 치명적인 문제를 발견하고 원인 분석을 요청.

**GEMINI:**
- 'Assertion' 버튼을 'Assertion Mode' 체크박스로 변경하고, 제안된 워크플로우에 맞춰 로직을 수정.
- Python의 검증 상태(`is_asserting`)를 `recorder.js`의 `window.isAsserting` 변수로 전달하는 로직을 `main.py`에 구현.
- `recorder.js`가 `window.isAsserting` 상태를 확인하여, `true`일 경우 클릭된 요소의 텍스트를 담은 `'assert_text'` 액션을, `false`일 경우 기존의 `'click'` 액션을 생성하도록 수정.
- 사용자가 지적한 클릭 이벤트 먹통 현상이 `event.preventDefault()` 때문임을 확인하고, 해당 코드를 제거하여 브라우저의 정상적인 인터랙션과 검증 스텝 기록이 동시에 가능하도록 문제를 해결함.
